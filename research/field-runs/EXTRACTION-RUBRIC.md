# 필드 런 추출 루브릭 — Codex 세션 JSONL 분석용

> 용도: 실전 루프 세션 기록(JSONL)을 Opus 서브에이전트에게 주고 인사이트만 추출할 때의 표준 지시.
> 원문은 메인 컨텍스트에 싣지 않는다. 산출물은 `research/field-runs/<날짜>-<슬러그>.md`.

## 추출 대상 (우선순위순)

### A. 루프 구조 관찰 (표준 검증 — 최우선)
1. **실행 형태 판별**: looprun/LOOP.md 기반인가, 아니면 비통제 자유 세션인가? (후자여도 귀중 — "야생 루프" 대조군)
2. **반복 구조**: 실질 iteration 수, 각 iteration의 단위 작업 크기, 방향 전환(Δ) 횟수와 계기
3. **판정 방식**: 검증이 독립 실행이었나(명령/테스트), 아니면 에이전트 자기보고였나? 자기보고가 판정에 쓰인 사례 발췌 (스펙 §4.2.5 위반 패턴의 실전 증거)
4. **정지**: 어떻게 끝났나 — 목표 충족 판정? 사용자 개입? 상한? 정체? (비통제 세션이라면: 상한 없이 얼마나 돌았고 누가 멈췄나)
5. **판정 자산**: 테스트/검증 스크립트를 에이전트가 수정한 순간이 있나 (있다면 발췌 — node 114 패턴의 실전 사례)

### B. 마찰·UX 관찰
6. 사용자가 개입해야 했던 모든 순간과 이유 (인간 게이트가 필요했던 지점의 실전 데이터)
7. 에이전트가 헤맨 구간 (같은 실패 반복, 컨텍스트 소실, 잘못된 방향 고집) — stagnation 감지가 있었다면 잡혔을 지점
8. 지시(프롬프트/LOOP.md)의 어느 부분이 오해됐나 — 저작 가이드 개선 입력

### C. 경제성 데이터 (n=1, loop-economics 엔티티 보탬용)
9. 토큰/비용/시간/모델, iteration당 평균, 낭비 구간(효과 없는 iteration) 비율
10. 반복 재사용 가능한 지식 후보 — 이 세션에서 플레이북으로 정제할 만한 패턴 3~5개 (§7.5 스키마 형태로)

### D. 앱 관점 (부차)
11. 대시보드 작업 자체의 결과 요약 3줄 + 남은 일 목록

## 출력 형식

한국어. 각 항목에 근거 발췌(라인 범위 인용, 최대 3줄)를 달 것. 추측과 관찰을 구분 표기. 마지막에 "스펙/스킬에 반영할 구체 제안" 3개 이내.

## 실행 메모

- 서브에이전트: Opus 4.8, 파일 경로만 전달 (원문 붙여넣기 금지)
- JSONL이 수 MB면: 에이전트에게 grep/jq로 구조 파악 후 선택적 정독 지시 (전체 Read 금지)
- 민감정보(회의 내용 등)가 있을 수 있음: 산출물에 앱 도메인 데이터 원문 인용 금지, 구조 관찰만
