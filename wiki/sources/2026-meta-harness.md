# [논문] Meta-Harness: End-to-End Optimization of Model Harnesses

- **URL:** https://arxiv.org/html/2603.28052v1
- **저자:** Yoonho Lee, Roshen Nair, Qizheng Zhang, Kangwook Lee, Omar Khattab, Chelsea Finn (Stanford, MIT, KRAFTON)
- **접근일:** 2026-07-06
- **유형:** arXiv 논문

## Digest

**주장:** LLM 성능은 가중치만이 아니라 **하네스**(무엇을 저장·검색·제시할지 결정하는 코드)에 달려 있다. 기존 텍스트 최적화(프롬프트 최적화 등)는 피드백을 스칼라 점수·짧은 요약으로 극도 압축해("memoryless") 하네스 엔지니어링에 부적합.

**루프 구조:**
1. **제안:** 코딩 에이전트(Claude Code, Opus-4.6)가 파일시스템에 저장된 **모든 이전 후보의 소스코드·실행 추적·점수**에 접근 (grep/cat으로 선택적 진단)
2. **평가:** 제안된 하네스를 평가 작업에서 테스트 — 단일 평가가 최대 1천만 토큰의 진단 정보 생성
3. **기록:** 새 디렉터리에 제안 코드·추론 추적·점수 저장
- 진행 추적: 이전 후보 간 **Pareto 지배 관계** 유지 (스칼라 단일 축이 아님)
- 예산: 통상 ~60개 하네스, 20회 반복

**결과:**
- 텍스트 분류: ACE 대비 +7.7pt, 컨텍스트 토큰 1/4. OOD 9개 평균 73.1% (ACE 70.2%)
- 수학 추론: IMO급 200문제에서 비검색 대비 +4.7pt, **본 적 없는 모델 5개에 전이**
- TerminalBench-2: Haiku 4.5에서 전체 에이전트 1위(37.6%), Opus 4.6에서 수동 설계 Terminus-KIRA 초과(76.4% vs 74.7%)

**인과 추론 사례 (부록 A.2):** 6회 연속 회귀 후 에이전트가 로그에서 원인 분리 — "회귀의 근본 원인: 프롬프트 템플릿의 cleanup 지시가 작업 완료 전 필요한 상태를 삭제하게 함" → 구조 개선과 프롬프트 재작성을 분리, 안전한 추가(additive) 수정 전략으로 전환.

**한계:** 단일 제안자(Claude Code)만 검증, TerminalBench-2 과적합 우려(regex 감시로 누출 확인), 평가 예산이 주요 제약, 무한 코드 탐색 공간은 에이전트 귀납 편향에 의존.
