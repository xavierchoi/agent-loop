# LOOP.md UX 시나리오 — 가상 사례로 돌려보는 전체 플로우

> 가상 사례 기반 시뮬레이션. 모든 수치는 **예시용(illustrative)**이며 실측이 아님.
> 스펙 근거: `design/loop-mechanics.md` (폴더 규격·라이프사이클·런타임 계약)

---

## 등장 설정

- **지민** — 3인 웹 에이전시 운영. 고객사 사이트 유지보수가 주 수입원.
- **루프**: `lighthouse-perf-optimizer` — "핵심 페이지들의 Lighthouse Performance ≥ 90을, 시각적 회귀 없이 달성·유지"
- **런타임**: LOOP.md Level 2 준수 CLI(가칭 `loop`). 에이전트 백엔드는 아무 코딩 에이전트나.
- **고객**: A(첫 고객, 리테일), B(두 번째, 미디어), C(세 번째, SaaS 랜딩).

이 사례를 고른 이유: 검증이 명령 실행으로 완전 독립(`lighthouse-ci`), 홀드아웃이 자연스럽고(페이지 목록 분할), 반복 패턴이 실재하며(이미지·폰트·서드파티 스크립트), 고객 간 전이가 일어나는 도메인 — 즉 스펙의 모든 필드가 실전에서 발화된다.

## 시나리오 0 — 저작과 설치 (Day 0): "목표 협상"

지민이 마켓플레이스에서 비슷한 루프를 발견하거나, 처음부터 만든다. 어느 쪽이든 **기동 전에 런타임이 필수 3필드를 인터뷰로 확정**한다 — 당신 다이어그램의 "목표 협상" 단계가 이것이다.

```
$ loop init lighthouse-perf-optimizer

◆ 목표 협상 (goal은 이후 불변 코어가 됩니다)
  ? 이 루프가 반복 달성할 목표는:            > 핵심 페이지 Lighthouse Perf ≥ 90
  ? 목표 달성을 판정할 독립 검증 방법은:      > npx lhci autorun --config=.lighthouserc
  ? 시각 회귀 게이트 추가? (권장)             > yes — playwright 스크린샷 diff < 0.1%
  ? 정지 조건: 최대 반복 / 완료 신호 / 정체    > 15회 / "2회 연속 all-green" / 3회 무진전
  ? 홀드아웃: 검증 페이지 12개 중 몇 개를
    개선 에이전트에게 숨길까요?               > 4개 (URL은 state/holdout/에 격리)
  ? 예산 상한:                              > $30 / 6시간

✔ LOOP.md v0 생성됨 (gene: a41f00)
✔ state/ 초기화, holdout ACL 설정 (improver-access: none)
```

**포인트:** goal·verify·holdout은 여기서 못 박히고 이후 진화 루프조차 못 건드린다(불변 코어). 사용자가 협상하는 건 "무엇을 성공이라 부를 것인가"이지, 에이전트가 그걸 어떻게 달성하는가(자유 변이 구역)가 아니다.

## 시나리오 1 — 첫 실행: 고객 A, 반복 12회 (LLM이 대부분을 담당)

```
$ loop run lighthouse-perf-optimizer --workdir ~/clients/a-retail
▶ 백그라운드 기동 (gene v0 · iteration 0/15 · budget $30)
```

지민은 다른 일을 한다. 런타임이 iteration을 돌린다 — **매 iteration은 신선한 컨텍스트로 시작하고, 기억은 전부 파일(로그·플레이북)이 담당한다.**

```
[iter 1] 플레이북 조회: 0건 매칭 (비어 있음) → LLM 구현 모드
         분석: LCP 4.2s. 히어로 이미지 1.8MB png → webp 변환 + priority hint
         verify ✗ Perf 71→78 (미달) · 시각 diff 0.0%
         이유 분석: 폰트 로딩이 차단 요인 → 방향: 폰트 최적화     [Δ 기록]
[iter 2] font-display:swap + preload … verify ✗ 78→84
[iter 3] 서드파티 채팅 위젯 지연 로딩 … verify ✗ 84→87
[iter 4] JS 번들 코드 스플리팅 … verify ✗ 87→89 (아깝게 미달)
[iter 5] 이미지 srcset 정리 … verify ✓ 91 — 1회차 green
[iter 6] (변경 없음, 재검증) verify ✓ 92 — 2회 연속 green ✔ 완료 신호
```

여기서 중요한 UX 계약 두 가지:

1. **자기보고 금지가 눈에 보인다.** iter 4에서 에이전트 세션은 "코드 스플리팅 완료, 이제 90 넘을 것"이라고 썼지만, 로그에 남는 판정은 오직 `verify` 프로세스의 89점이다. 사용자는 에이전트의 낙관이 아니라 검증기의 숫자만 보게 된다.
2. **산출물에는 "왜 성공" 데이터가 붙는다.** 종료 시:

```
✔ lighthouse-perf-optimizer 완료 — 고객 A
  12 iterations · $19.40 · 4h 12m
  산출물: PR #47 (12 커밋, 시각 diff 0%)
  왜 성공: [이미지 webp+priority, 폰트 preload, 서드파티 지연, 코드 스플릿] 기여도 로그
  홀드아웃 검증: 숨겨진 4페이지 평균 90.5 ✓ (과적합 아님)
  원시 로그: state/logs/raw.jsonl (+2,847줄, append-only)
```

**홀드아웃의 첫 발화:** 개선이 에이전트가 본 8페이지에만 통했는지, 안 본 4페이지에도 일반화됐는지를 종료 시점에 같은 잣대로 확인한다. (에이전트가 8페이지의 특정 DOM에만 하드코딩 최적화를 했다면 여기서 걸린다 — WebArena 하드코딩 과적합 사례와 동형.)

## 시나리오 2 — 진화 루프 발동: v0 → v1 (증류 + 래칫 + 인간 게이트)

고객 A 건이 2회 더 돌아간 뒤(정기 유지보수), 런타임이 진화 주기를 발동한다. **개선 에이전트는 구현 에이전트와 다른 세션이고, state/holdout/에 접근할 수 없다(파일시스템 ACL).**

```
$ loop evolve lighthouse-perf-optimizer        # 또는 3-run마다 자동
◆ 증류: 3개 런 · 8,412줄 로그 → 후보 패턴 4건
  1. "png/jpg 히어로 → webp + fetchpriority" (3/3 런에서 유효)  → 휴리스틱 등록
  2. "font-display:swap + preload woff2"      (3/3)            → 휴리스틱 등록
  3. "서드파티 위젯 → facade 패턴"             (2/3)            → 휴리스틱 등록
  4. "고객 A CMS의 이미지 필드 특이 처리"       (A에서만 관측)    → playbook/local/a-retail/ (격리)

◆ 유전자 변이 제안 (gene v0 → v1):
  [자동 승인 · free]    운영 지침 §반복당 절차에 "이미지→폰트→서드파티 순서로 점검" 추가
  [자동 적용 · ratchet] pass-threshold 90 → 91  (3런 연속 여유 통과 → 단방향 강화)
  [게이트 대기 · gated] stop.max-iterations 15 → 10 제안
                        근거: "최근 3런 평균 7회에 수렴"
```

지민에게 알림이 온다 — **게이트 승인 카드:**

```
┌─ 인간 게이트: lighthouse-perf-optimizer ─────────────────┐
│ 제안: 최대 반복 15 → 10 (정지조건 변경 = gated 항목)      │
│ 근거: 최근 3런 평균 7.3회에 완료                          │
│ 홀드아웃 리허설: gene v1, 숨긴 4페이지에서 v0와 동일 잣대   │
│   → v0: 90.5 / v1: 90.8 (성능 저하 없음)                 │
│ [승인] [거부] [수정…]                                     │
└──────────────────────────────────────────────────────────┘
```

**포인트:** 강화(threshold 90→91)는 물어보지 않고 자동이다. 완화·구조 변경만 사람을 부른다. 그리고 신구 유전자는 반드시 **같은 홀드아웃, 같은 잣대**로 비교된 결과가 카드에 첨부된다. 승인하면 LOOP.md가 수정·커밋되고 git 히스토리에 v1이 남는다.

## 시나리오 3 — 고객 B: 반복 5회 (라우터가 일을 나눠 갖기 시작)

```
$ loop run lighthouse-perf-optimizer --workdir ~/clients/b-media
[iter 1] 플레이북 조회: 휴리스틱 3건 매칭
         → LLM이 휴리스틱을 참조하며 구현 (이미지·폰트 한 번에)
         verify ✗ 68→83 (한 iteration에 15점 — A 때 4 iteration 분량)
[iter 2] 서드파티: facade 휴리스틱 적용 … verify ✗ 83→88
[iter 3] B 특유의 무한 캐러셀이 LCP 차단 — 플레이북 무매칭 → 순수 LLM 탐구
         verify ✗ 88→89
[iter 4] 캐러셀 첫 슬라이드 SSR 고정 … verify ✓ 91
[iter 5] 재검증 ✓ 92 — 완료
✔ 5 iterations · $8.10 · 1h 40m   (A: 12회·$19.40 대비)
```

- 축적된 휴리스틱 덕에 **탐색이 아니라 적용**으로 시작 → 반복 수 급감. (Voyager의 스킬 라이브러리 제로샷 전이와 같은 메커니즘)
- 캐러셀 건은 새 지식 → 로그에 남고 다음 증류 때 후보가 된다.
- A의 CMS 특이 처리는 `playbook/local/a-retail/`에 격리돼 있으므로 **B 실행의 컨텍스트에 아예 로드되지 않는다** — 고객 격리가 "지시"가 아니라 디렉터리 스코핑으로 구현.

## 시나리오 4 — 승격과 강등: 휴리스틱 → 규칙 → 공식

두 번째 진화 주기. 승격 사다리가 움직인다:

```
◆ 승격 심사
  "이미지 webp+priority" 휴리스틱
    · 에피소드 5회 유효 (k=5 충족) → 규칙으로 승격
  "폰트 preload" 규칙
    · 규칙 승격 후 4회 무실패 + 홀드아웃 통과 → 공식으로 승격
    → scripts/font-optimize.mjs 로 컴파일 (codemod, 결정론)
    → 라우터 등록: 전제조건 [woff2 존재 · Next.js ≥ 14] 충족 시 LLM 없이 실행
```

이제 다음 실행부터 폰트 최적화는 **토큰을 한 푼도 안 쓰는 스크립트 실행**이다(SKILL.md scripts와 동형 — 코드는 컨텍스트에 로드되지 않고 출력만 소비).

몇 주 뒤, 강등도 일어난다:

```
[고객 A 정기 런] scripts/font-optimize.mjs 실행 → verify ✗ 회귀 감지 (Next 15 폰트 파이프라인 변경)
◆ 자동 강등: 공식 → 규칙 (라우터에서 제거, LLM 폴백 복귀)
  로그: 강등 사유 + 실패 케이스 → 다음 진화 주기의 수선 후보
```

**포인트:** 강등은 자동이고 즉각적이다(검증이 잡으니까). 공식의 지위는 영구 신분이 아니라 "홀드아웃을 계속 통과하는 동안"의 계약이다.

## 시나리오 5 — 새 모델 출시: 재검증 플래그

에이전트 백엔드 모델이 업데이트되면 런타임이 플래그를 세운다:

```
⚑ 모델 변경 감지 (claude-x → claude-y)
◆ 재검증: 현행 gene v2를 홀드아웃 전체에 리허설
  · 공식(scripts/) 경로: 영향 없음 — 결정론 실행은 모델 무관 ✓
  · LLM 폴백 경로: 2케이스 재실행 → 1건 미묘한 품질 변화 → 다음 증류 때 주시 태그
```

**여기가 LOOP.md의 조용한 세일즈 포인트다:** 결정론 비율이 높아질수록 모델 교체 리스크가 줄어든다. 축적된 가치의 대부분이 "특정 모델의 습관"이 아니라 실행 가능한 공식에 응고돼 있기 때문이다. (2차 리서치의 "구조 > 지능" 방향 시사와 연결)

## 시나리오 6 — 고객 C(반복 2회)와 성숙: 루프가 스킬을 낳다

```
$ loop run lighthouse-perf-optimizer --workdir ~/clients/c-saas
[iter 1] 라우터: 공식 3건 전제조건 충족 → 결정론 실행 (이미지·폰트·서드파티)
         잔여: 없음에 가까움 → LLM 미세 조정 1건
         verify ✗ 74→89
[iter 2] 마무리 → verify ✓ 92 → 재검증 ✓ — 완료
✔ 2 iterations · $2.30 · 25m
```

**경제 곡선(예시 수치):** A 12회·$19.40 → B 5회·$8.10 → C 2회·$2.30. 결정론 비율 0% → ~40% → ~80%. — 인포그래픽의 두 그래프가 이 시점에 실데이터로 그려진다. (주의: 2차 리서치대로 이 곡선은 아직 공개 실증이 없는 **검증 대상 가설**이다. 런타임이 이 곡선을 로그에서 자동 산출해 보여주는 것 자체가 표준의 킬러 기능이 될 수 있다 — 모든 인스턴스가 자기 경제 곡선의 증거를 생산한다.)

성숙의 마지막 단계:

```
$ loop export-skill lighthouse-perf-optimizer
✔ skills/lighthouse-fix-pack/ 생성
  SKILL.md (name, description ← 루프 frontmatter에서 파생)
  scripts/ ← 승격된 공식 3건
→ 이제 루프 없이도, 아무 Agent Skills 호환 에이전트가 이 공식을 쓸 수 있다

$ loop publish lighthouse-perf-optimizer
◆ 배포 패키징: LOOP.md + playbook/generic/ + scripts/
  제외: state/* · playbook/local/* (매니페스트 규칙)
  마켓플레이스 카드: goal · verify 방식 · 누적 홀드아웃 통과 이력 (신뢰 신호)
```

다른 에이전시가 이 루프를 설치하면 **지민의 유전자 + 범용 플레이북에서 시작**한다 — 그들의 "고객 A"는 12회가 아니라 4~5회부터 시작할 것이다. 이것이 마켓플레이스의 실제 가치 제안이다: 프롬프트를 파는 게 아니라 **수렴이 끝난 유전자를 판다.**

## UX 표면 총정리

| 표면 | 언제 | 형태 |
|---|---|---|
| `loop init` 인터뷰 | 저작 | 목표 협상 CLI 위저드 (필수 3필드 확정) |
| `loop run` | 기동 | 백그라운드 잡 + 진행 라인 (iter n/N · verify 결과 · $) |
| `loop status` / 대시보드 | 상시 | 루프별: 현재 gene 버전 · 경제 곡선 · 플레이북 사다리 현황 |
| 게이트 승인 카드 | 완화·구조 변경 제안 시 | 제안 + 근거 + **홀드아웃 신구 비교** 첨부, 승인/거부/수정 |
| 완료 리포트 | 종료 | 산출물 + "왜 성공" + 홀드아웃 일반화 확인 |
| ⚑ 재검증 알림 | 모델 변경 시 | 자동 리허설 결과 요약 |
| 강등 알림 | 공식 회귀 시 | 자동 조치 통보 (사후 통보로 충분 — 안전 방향 변화이므로) |

**알림 설계 원칙:** 사람을 부르는 것은 오직 ① 완화/구조 변경 게이트, ② 예산·정체 킬스위치 발동, ③ 완료. 강화·승격·강등은 안전한 방향의 변화이므로 사후 통보다. — "인간의 주의"를 가장 희소한 자원으로 취급하는 것이 래칫 비대칭의 UX적 의미다.

## 실패 UX (해피 패스만큼 중요)

- **정체 정지:** 3회 무진전 → 루프가 스스로 멈추고 "시도한 방향 3가지 + 막힌 지점" 요약과 함께 사람에게 돌려준다. (self-improvement reversal의 4–5회 포화 관측이 근거)
- **예산 킬스위치:** $30 도달 → 즉시 정지, 부분 산출물 + 로그 보존. 다음 기동은 이어달리기(로그가 기억이므로).
- **게이트 방치:** 승인 대기 72h 초과 → 루프는 v(N) 유전자로 계속 운행(제안은 큐에 유지). 진화가 막혀도 운영은 멈추지 않는다.
- **검증기 자체의 오염 의심:** verify 결과와 홀드아웃 결과가 계속 벌어지면(과적합 신호) 런타임이 경고 — Goodhart 곡선의 조기 감지.
