## 1. 개요

**Agent Loop**(이하 루프)는 AI 에이전트가 **선언된 목표를 독립 검증이 통과할 때까지 반복 시도**하도록 지시하는, 파일시스템 기반의 이식 가능한 패키지다.

루프는 `LOOP.md` 파일을 담은 디렉터리다. `LOOP.md`는 YAML frontmatter(목표·정지조건·검증 선언)와 마크다운 본문(운영 지침)으로 구성된다.

### 1.1 SKILL.md와의 관계

| | Agent Skills | Agent Loops |
|---|---|---|
| 선언하는 것 | 능력 — *무엇을 할 줄 아는가* | 과정 — *어떤 목표를, 언제까지 반복하며, 무엇으로 판정하는가* |
| 상태 | 무상태 (읽고 버림) | 유상태 (`state/`에 실행 이력 축적) |
| 트리거 | 모델 주도 (description 매칭) | 명시적 기동 (사용자 · 스케줄 · 상위 에이전트) |
| 자기 변경 | 없음 | 선택적 (Evolution 확장, §7) |

루프는 스킬을 **소비**한다(§4.2.9 `skills`). 루프가 배양한 결정론 실행물은 스킬로 **내보낼 수** 있다(§8.3). 기존 루프 관행(PROMPT.md/Ralph)·워크플로 엔진과의 관계는 부록 D 참조.

### 1.2 설계 원칙

1. **최소 필수** — 필수 필드는 5개뿐이다. 그 외 전부 선택.
2. **자기보고 금지** — 구현 주체의 달성 선언은 판정에 사용되지 않는다. 판정은 오직 선언된 검증기가 하며, 검증기가 참조하는 판정 자산은 구현 주체가 변경할 수 없어야 한다(§4.2.5).
3. **점진적 공개** — frontmatter만 상시 로드 대상이며, 본문·플레이북·로그는 필요 시 파일시스템에서 읽는다.
4. **점진적 준수** — 런타임 없이 문서로도 유효하고(Level 0), 런타임 기능은 수준별로 쌓아 올린다(§6).
5. **정의/인스턴스 분리** — 배포되는 것은 정의(**유전자(gene)**, 특정 버전의 LOOP.md 내용)이고, 실행 상태는 인스턴스에 남으며 절대 배포되지 않는다(§8).

### 1.3 30초 요약

유효한 최소 루프는 아래가 전부다 — 이것을 아무 에이전트에게 주면 "테스트가 통과할 때까지, 스스로 다 됐다고 우기지 말고, 최대 15회까지 고쳐라"가 된다.

```yaml
---
name: fix-regressions
description: 회귀 테스트가 통과할 때까지 고친다. 야간 또는 대형 머지 후 사용.
goal: main 브랜치의 모든 회귀 테스트 통과
stop: { max-iterations: 15 }
verify: { run: "npm test" }
---
실패 테스트를 하나씩 고친다. 완료 판정은 verify가 한다 — 스스로 선언하지 말 것.
```

나머지 필드와 §7 Evolution 확장은 전부 이 뼈대 위의 선택 사항이다.

## 2. 디렉터리 구조

```
loop-name/
├── LOOP.md              # 필수. 루프 정의 (유전자)
├── scripts/             # 선택. 결정론 실행물 (실행 코드)
├── playbook/
│   ├── generic/         # 선택. 이식 가능한 정제 지식
│   └── local/           # 선택. 인스턴스/고객 전용 지식 — 배포 금지
└── state/               # 런타임 생성. 인스턴스 상태 — 배포 금지
    ├── logs/            # append-only 실행 로그
    ├── genes/           # LOOP.md 버전 스냅샷 (git 미사용 환경용)
    ├── holdout/         # 홀드아웃 자산 (Evolution 확장, 접근 통제 구역)
    ├── ratchet/         # 메타레이어 강화 이력 (Anti-Goodhart, §7.7. 선택)
    └── HANDOFF.md       # 크로스-세션 연속성 요약 (선택, §5.7)
```

- 루프는 최소한 `LOOP.md` 파일 하나를 담은 디렉터리다. 다른 모든 항목은 선택이다.
- 디렉터리명은 frontmatter의 `name`과 일치해야 한다(MUST). 단일 파일 공유(gist 등)는 Level 0에서 이 일치 요구를 면제한다(§6).
- `state/`와 `playbook/local/`은 런타임/운영 중 생성되며, 배포 패키지에 포함해서는 안 된다(MUST NOT, §8.1).
- **작업 대상(target):** 루프가 실제로 편집·검증하는 대상(예: 고칠 코드베이스)은 루프 디렉터리와 별개다. 둘의 관계는 §5.6에서 정의한다.
- `state/HANDOFF.md`는 크로스-세션 인수인계 요약을 위한 **선택 산출물**이다(§5.7). 나머지 `state/` 하위 트리와 마찬가지로 배포 금지(MUST NOT, §8.1)이며, 자유형 서사이므로 어떤 판정에도 입력되지 않는다(자기보고 금지 §1.2 원칙 2).

## 3. LOOP.md 파일 포맷

```
---
(YAML frontmatter)
---
(마크다운 본문)
```

- 파일은 유효한 YAML frontmatter로 시작해야 한다(MUST).
- **YAML 기준 (v0.4, 근거: 백로그 F1):** frontmatter는 **YAML 1.2 core schema**로 해석해야 한다(MUST). 이는 상호운용의 기준선이다 — 서로 다른 러너가 같은 `LOOP.md`를 같은 타입으로 읽으려면 스칼라 판정(불리언·널·정수·실수·문자열)의 규칙이 고정되어야 한다. 런타임이 YAML의 **서브셋만** 지원하는 경우(예: 앵커·태그·복수 문서 미지원), 지원 범위와 미지원 시 동작(거부/무시)을 문서화해야 한다(MUST). 작성자는 이식성을 위해 core schema 범위 안에서 작성할 것을 권장한다(SHOULD).
- 본문은 자유 형식 마크다운이다. 형식 제약은 없으나 500줄 미만을 권장한다(SHOULD).
- 본문에서 참조하는 파일은 루프 디렉터리 기준 상대 경로여야 한다(MUST).

## 4. Frontmatter 필드

### 4.1 필드 요약표

| 필드 | 필수 | 타입 | 제약 |
|---|---|---|---|
| `name` | **Yes** | string | ≤64자. 소문자·숫자·하이픈. 하이픈 시작/끝/연속 금지. 디렉터리명과 일치(§4.2.1) |
| `description` | **Yes** | string | 비어있지 않은 ≤1024자. "무엇을 반복 달성 + 언제 쓰는지" 모두 기술 |
| `goal` | **Yes** | string | 비어있지 않은 ≤1024자. **불변** (§4.2.3) |
| `stop` | **Yes** | object 또는 int | §4.2.4의 상한 계열 최소 1개 포함. `stop: 15` = `{max-iterations: 15}` |
| `verify` | **Yes** | object · object[] · string | §4.2.5. `verify: "npm test"` = `{run: "npm test"}` |
| `spec` | No | string | 이 파일이 준수하는 명세 식별자 (예: `agent-loops/0.5`). §4.2.10 |
| `budget` | No | object | §4.2.6 |
| `log` | No | object | 기본값 있음. §4.2.7 |
| `holdout` | No† | object | Evolution 확장. §7.2 |
| `mutable` | No† | object | Evolution 확장. §7.3 |
| `evolve` | No | `auto`\|`manual`\|`off` | Evolution 확장. run 후 정제 실행 정책. 미선언 시 런타임 기본 `auto` 권장. §7.1 |
| `skills` | No | string[] | 소비하는 Agent Skills의 name 목록 |
| `license` | No | string | SPDX 식별자 권장 |
| `compatibility` | No | string | ≤500자. 요구 런타임·도구·환경·능력(egress·fs 범위) |
| `metadata` | No | map | 임의 키-값. 스펙은 해석하지 않음 |

† `holdout`·`mutable`은 선택이지만, **선언된 경우** Level 2 런타임은 이를 강제해야 한다(MUST). §6.

**단축 표기:** `stop`과 `verify`는 스칼라 단축을 허용한다(위 표). 작성 진입 장벽을 낮추기 위한 것으로, 런타임은 이를 정규 객체형으로 확장해 해석해야 한다(MUST).

### 4.2 필드 정의

#### 4.2.1 `name`
최대 64자, 소문자·숫자·하이픈만 사용하며, 하이픈으로 시작·끝나거나 연속될 수 없고, 부모 디렉터리명과 일치해야 한다(MUST). (이 규칙은 Agent Skills의 `name` 규칙과 동일하나, 자기완결성을 위해 여기 인라인한다 — 외부 명세에 규범 의존하지 않는다.)

#### 4.2.2 `description`
루프가 **무엇을 반복 달성하는지**와 **언제 쓰는지**를 모두 기술해야 한다(MUST). 카탈로그·마켓플레이스 노출과 상위 에이전트의 루프 선택에 사용되므로, 매칭에 도움이 되는 구체적 키워드를 포함해야 한다(SHOULD).

#### 4.2.3 `goal`
루프의 최상위 목표. 다음이 성립한다:

- `goal`은 **불변 코어**다. Evolution 확장을 포함한 어떤 자동 프로세스도 `goal`을 수정해서는 안 되며(MUST NOT), 런타임은 이를 강제해야 한다(MUST).
- `goal`의 변경은 새 루프의 작성 또는 인간의 직접 편집으로만 가능하다.
- 검증 가능한 형태로 서술해야 한다(SHOULD). 나쁜 예: "사이트를 빠르게". 좋은 예: "핵심 페이지 Lighthouse Performance ≥ 90, 시각 회귀 0".

#### 4.2.4 `stop` — 정지조건

| 하위 필드 | 타입 | 의미 |
|---|---|---|
| `max-iterations` | int ≥ 1 | 반복 횟수 상한 |
| `max-duration` | duration | 벽시계 시간 상한 (예: `"6h"`, `"90m"`). 원점·계측은 §4.2.6 |
| `green-runs` | int ≥ 1 (기본 1) | 완료 판정에 필요한 **연속 검증 통과** 횟수 |
| `stagnation` | int ≥ 1 | 연속 무진전 반복 횟수 도달 시 `stalled` 종료 |
| `progress-paths` | string[] | 선택. 무진전 판정 시 해시 대상으로 삼을 산출물 경로. 미선언 시 기본은 작업 대상(target) 트리 전체(§4.2.4 제외 규칙). |

- `stop`은 `max-iterations` 또는 `max-duration` 중 **최소 하나**를 포함해야 한다(MUST). 무한 루프의 선언을 스펙이 허용하지 않는다. (상시 운영은 §5.2의 세션-재기동 패턴으로 표현한다.)
- **green-runs 리셋 규칙:** 다음 중 어느 iteration이라도 발생하면 green-runs 연속 카운터를 0으로 리셋한다(MUST): 검증 실패, 검증 미실행(구현 주체 오류·예산 kill 등), 부분 통과(배열 검증에서 일부 실패). 즉 "연속"은 완전 통과 iteration의 무중단 연쇄를 뜻한다.
- **stagnation 기본 알고리즘:** `verify.score`가 있으면 무진전 = "best-so-far 점수 대비 개선 없음"(개선 임계 기본 0, 루프가 override 가능). 점수가 없으면 무진전 = "verify 결과(통과/실패 및 출력)가 직전과 동일하고, **진행도 해시 대상 경로**의 해시 집합에 변화 없음". 런타임이 이 기본과 다른 판정을 쓰면 그 기준을 문서화해야 한다(MUST).
- **진행도 해시 대상 (v0.4, 근거: 백로그 감사관 항목):** 무점수 stagnation의 해시 대상은 `stop.progress-paths`로 선언할 수 있다(선택). 선언 시 런타임은 그 경로 집합만 해시해야 한다(MUST). **미선언 시 기본값은 작업 대상(target §5.6) 트리 전체**이며, 이때 버전관리·의존성 캐시 등 파생·무관 디렉터리(예: `.git`, `node_modules`)는 해시에서 제외해야 한다(MUST) — 그렇지 않으면 커밋 메타데이터·설치물의 잡음이 거짓 진행으로 오판된다. (이 기본값은 레퍼런스 러너 `runtime/looprun`의 관례를 표준화한 것이다: verify 출력 + target 트리 다이제스트, `.git`/`node_modules` 제외.)
- **진행도 해시와 판정 자산의 분리 (v0.5, 근거: v0.4 열린 질문 #3 → MUST 승격):** 진행도 해시 대상은 판정 자산(`verify.assets`, §4.2.5) 경로와 겹치지 않아야 한다 — 작성자는 `progress-paths`를 `assets`와 겹치지 않게 선언해야 하고(SHOULD), **런타임은 실효 진행도 해시 대상에서 `verify.assets`로 선언된 경로를 배제해야 한다(MUST)**. 근거: 판정 자산은 매 검증 전 격리 복원되어 해시가 상시 불변이므로, 이를 진행도 대상에 포함하면 실제 진전을 상쇄해 거짓 정체(stalled)를 유발한다. v0.4에서 SHOULD 지침으로 부분 처리했으나, 기본값(미선언 시 target 트리 전체)이 자연히 assets를 포함하는 흔한 경우를 SHOULD만으로는 막지 못하므로 런타임의 자동 배제를 MUST로 승격한다. (작성자의 명시 `progress-paths`가 그럼에도 assets를 포함하면, 런타임은 그 교집합을 배제하고 이를 로그에 남긴다(SHOULD).)
- 종료 상태는 다음 중 하나여야 한다(MUST): `completed`(green-runs 충족) · `stalled` · `max-iterations` · `max-duration` · `budget-exceeded` · `aborted`(인간 중단).

#### 4.2.5 `verify` — 독립 검증

| 하위 필드 | 타입 | 의미 |
|---|---|---|
| `run` | string | 셸 명령. 종료 코드 0 = 통과 |
| `agent` | string (경로) | 비평가 에이전트 지시문 파일. 구현과 **분리된 컨텍스트**에서 실행 |
| `n` | int ≥ 1 (기본 1) | 선택. `agent` 검증기의 독립 실행 횟수 (앙상블) |
| `consensus` | `all` \| `majority` (기본 `all`) | 선택. `n` > 1일 때 통과 판정 규칙 |
| `score` | object | 선택. 진행도 점수. `{ command \| pattern, direction: higher-better\|lower-better }` |
| `assets` | string[] | 선택. 이 검증기가 참조하는 **판정 자산** 경로 (테스트·채점 스크립트·기대값). 격리 대상 선언. §4.2.5 |

- `verify`는 `run` 또는 `agent` 중 하나 이상을 포함해야 한다(MUST). 배열로 선언 시 **모든** 검증기가 통과해야 통과다(MUST).
- **배열 의미론:** 배열의 검증기는 선언 순서대로 **전부 실행**한다(로깅 목적, 단락 평가 금지). 하나라도 실패면 그 iteration은 검증 실패다. 여러 검증기가 `score`를 낼 때 stagnation·비교에 쓰는 **권위 있는 점수**는 `score`를 가진 첫 검증기(또는 명시 지정된 것)의 값이다(MUST).
- **`score` 방향·형식:** `score`는 방향(`higher-better`/`lower-better`)을 반드시 선언해야 한다(MUST) — 방향 없이는 "개선"의 부호를 판정할 수 없다. `command` 모드는 stdout 마지막 줄의 단일 실수를, `pattern` 모드는 첫 캡처 그룹을 점수로 해석한다. 둘은 혼용하지 않는다.
- **자기보고 금지(핵심 규칙):** 구현을 수행한 컨텍스트가 산출한 어떤 완료 선언·자체 평가도 판정에 사용해서는 안 된다(MUST NOT). `verify.agent`는 구현 컨텍스트의 대화 이력·중간 산출 요약·자체 평가가 입력에 포함되지 않는 새 세션/호출에서 실행해야 하며(MUST), 그 입력은 산출물(작업트리 스냅샷 또는 diff)과 `goal`, 검증 지시문 파일로 한정한다(SHOULD). 프롬프트 상의 역할 전환("너는 이제 비평가다")만으로는 준수가 아니다.
- **`agent` 검증기 판정 프로토콜 (v0.4, 근거: 백로그 F2):** `agent` 검증기의 **최소 계약**은 다음과 같다. 구체 실행 명령(어떤 CLI·모델로 지시문 파일을 실행하는가)은 런타임이 정의하되, 계약의 세 요소는 규범이다.
  - **입력(MUST):** 비평가 실행에 전달되는 입력은 (a) 산출물 스냅샷 또는 diff, (b) `goal`, (c) `verify.agent`가 가리키는 검증 지시문 파일로 한정한다. 위 자기보고 금지에 따라 구현 컨텍스트의 대화·자평은 포함하지 않는다(MUST NOT).
  - **판정 출력(MUST):** 비평가는 판정을 다음 중 한 형식으로 낸다 — **표준출력 마지막 줄이 `VERDICT: pass` 또는 `VERDICT: fail`**, 또는 **프로세스 종료 코드**(0 = pass, 그 외 = fail). 런타임은 자신이 어느 규약을 읽는지 문서화해야 한다(MUST). 둘 다 없으면 판정 불능이며, 판정 불능은 통과가 아니다(검증 미실행으로 간주 — green-runs 리셋, §4.2.4).
  - **형식 충돌 기본 우선순위 (v0.5, 근거: v0.4 열린 질문 #2):** 런타임이 두 규약을 모두 읽는데 서로 모순될 때(예: 마지막 줄 `VERDICT: pass`인데 종료 코드가 0이 아님), 기본값으로 **fail로 판정해야 한다(MUST)** — 즉 두 신호가 일치할 때만 pass. 근거: 안전 기본값은 자기보고 금지(원칙 2)의 정신과 정합하며, 위양성 통과(거짓 green)가 위음성보다 비용이 크다. 런타임이 단일 규약만 권위로 삼겠다고 문서화한 경우(예: "종료 코드만 읽음")에는 그 규약을 따르되, 그 선택을 문서화해야 한다(MUST).
  - **합의 집계(MUST):** `n` > 1일 때 런타임은 `n`회 독립 실행의 판정을 `consensus` 규칙으로 집계한다 — `all`은 전원 pass 시에만 pass, `majority`는 과반 pass 시 pass(동수는 fail). 각 실행은 서로의 출력을 입력으로 받지 않는 독립 호출이어야 한다(MUST).
- **판정 자산 선언 (`verify.assets`) (v0.4, 근거: 백로그 F4):** 검증기가 참조하는 판정 자산(테스트 파일·채점 스크립트·기대값)은 `assets`(작업 대상 기준 상대 경로 배열)로 선언할 수 있다. 이는 아래 격리의 **격리 대상을 명시**하는 선택 필드다.
- **판정 자산 격리 (v0.3, 강화 v0.4, 중요):** `verify.run`/`verify.agent`/`verify.score`가 참조하는 판정 자산은 구현 주체가 쓸 수 없어야 한다(MUST). 격리 수단은 다음 중 하나다: (a) `assets`로 선언된 경로를 런타임이 매 iteration 검증 전 불변 사본에서 복원, 또는 (b) 판정 자산을 clean-ref(예: git의 특정 커밋·별도 워크트리)에서 체크아웃해 구현 주체의 쓰기 표면 밖에서 실행. **Level 1 규칙(MUST):** `assets` 선언 또는 clean-ref 체크아웃 중 **어느 격리 수단도 확보되지 않으면 런타임은 실행을 거부해야 한다** — 격리 없는 실행을 Level 1로 표기해서는 안 된다(§6·§9.4). (이는 레퍼런스 러너 `runtime/looprun`의 거부 원칙 — `assets` 미선언 루프 실행 거부 — 을 표준화한 것이다.) 근거: 판정 자산이 구현 주체의 쓰기 표면 안에 있으면, 구현 주체가 테스트를 삭제·약화시켜 통과를 위조할 수 있다(DGM node 114 패턴). 이 격리 없이는 자기보고 금지(원칙 2)가 형해화된다.
- **파일 미만 단위 격리의 한계 (v0.4, 근거: 백로그 감사관 항목):** 위 격리 수단은 모두 **파일(또는 경로) 단위**로 동작한다 — 판정 자산과 가변 소스가 **같은 파일에 공존**하면 격리는 그 파일을 통째로 복원할 수밖에 없어, 구현 주체의 정당한 수정까지 되돌리거나(거짓 롤백) 판정 자산 변조를 놓친다. 이 명세는 파일 미만 단위 격리(AST·라인 범위 잠금)를 **규정하지 않는다** — 표준화는 향후 과제다. 대신 저작 지침으로 처리한다: **판정 자산과 가변 소스는 파일 단위로 분리해야 한다(SHOULD)** — 예로 테스트는 구현과 다른 파일·디렉터리에 두고, 채점 스크립트를 편집 대상 트리 밖에 배치한다.
- 근거(생성-검증 분리): 조사된 주요 자기개선 시스템 전부(Voyager·ADAS·AlphaEvolve·DGM·SICA)가 이를 채택했다.

##### 4.2.5a 프록시 품질 — 두 축의 구분 (v0.5, 근거: 필드 런 §3)

여기까지의 §4.2.5 규범(자기보고 금지·판정 자산 격리·판정 프로토콜)은 검증의 **축 A(실행 독립성)** 만 다룬다 — "판정이 구현 주체와 분리된 채널에서 돌고, 판정 자산이 변조 불가한가". 첫 실전 필드 런은 이것만으로 불충분함을 실증했다: 판정이 완전히 독립적(자산 변조 0, 명령·테스트 기반)이면서도 **측정 지표가 목표를 배신하는 프록시**일 수 있다 — 관측 사례에서 시각 점수 만점은 "무채색·상자 적음·금지어 없는 idle 화면"만 보장했고, 이는 "좋은 대시보드"라는 목표와 반대 방향이었다. 이를 검증의 **축 B(프록시 타당성)** 로 부른다: "그 독립 판정이 측정하는 지표가 실제 `goal`을 담보하는가". 축 A가 참이어도 축 B는 거짓일 수 있으며, 둘은 독립적이다.

축 B는 본질적으로 완전 형식화가 어렵다(프록시-목표 간극은 Goodhart의 법칙 그 자체다). 이 명세는 축 B를 **강제하지 않고**, 프록시 설계 시의 규범적 권고로 다룬다:

- **페널티-only 산식의 최적점 검사 (SHOULD):** `verify.score`·`verify.run`이 "나쁜 특징을 감점"하는 방식(금지어·상자 수·색 편차 페널티 등)으로만 구성되면, 그 산식의 **전역 최적점이 목표에 반하는 자명해**(예: 빈 화면·무채색·무내용)가 되지 않는지 검사해야 한다(SHOULD). 검사 방법: 자명한 열화 산출물(빈 결과·최소 산출물)을 판정기에 넣어 만점이 나오면 프록시가 배신 상태다. 페널티는 하한을 지킬 뿐 상한(품질)을 끌어올리지 못하므로, 목표가 "좋음"이면 **긍정 신호(밀도·구조·정량 존재 검사)를 함께 선언**할 것을 권장한다(SHOULD).
- **주관적(taste) 목표의 정량 게이트 (SHOULD, 강한 권고):** `goal`이 미적·주관적 품질(디자인·글의 톤 등)일 때, 검증을 자연어 자기 판정에만 맡기면 자기보고 금지(원칙 2)를 우회하기 쉽고 신뢰가 무너진다. 이런 목표는 **정량 게이트(예: 요소 밀도·구조 카운트·정량 임계를 건 DOM/AST 어서션)를 처음부터 함께 선언**해야 한다(SHOULD) — 필드 런에서 사용자가 "인간 게이트 없이" 시도했다가 자연어 taste 판정의 불신 때문에 정량 기준으로 회귀한 경로가 그 근거다. (이는 축 B를 완전 해결하지 않고, 프록시를 목표에 더 가깝게 조이는 실무 지침이다.)
- **정확문자열 blocklist 금지 (SHOULD, 근거: 필드 런 §6-4):** 판정에서 금지 조건을 **정확 문자열 일치(exact-match) blocklist**로 구현하는 것을 지양해야 한다(SHOULD) — 소수의 고정 문자열 대조는 패러프레이즈·동의어로 자명하게 우회된다(관측 사례: 4개 고정 문자열 가드가 표현 변경으로 뚫림). 대신 **의미 기반 판정**(임베딩·분류·반증 서브에이전트 `verify.agent`)을 권장한다(SHOULD). exact-match는 안정적 기계 신호가 필요한 좁은 경우(예: 특정 토큰의 물리적 부재 확인)로 한정한다.

이 절의 권고는 §7.7(Anti-Goodhart 메타레이어 강화)와 짝을 이룬다: 여기(축 B)가 "프록시를 처음부터 목표에 가깝게 설계"라면, §7.7은 "런을 거치며 프록시를 단조 강화"다.

#### 4.2.6 `budget`

| 하위 필드 | 타입 | 의미 |
|---|---|---|
| `max-cost-usd` | number | 누적 비용 상한 |
| `max-duration` | duration | `stop.max-duration`과 동일(중복 선언 시 짧은 쪽 우선) |
| `per-iteration` | duration 또는 object | 선택. **iteration당** 자원 상한 (예: `"5m"` 또는 `{ duration: "5m", cost-usd: 2 }`) |

- Level 1+ 런타임은 상한 도달 시 즉시 정지하고 로그를 보존해야 한다(MUST). 누적 상한 도달 시 종료 상태는 `budget-exceeded`.
- **비용 계측 범위:** `max-cost-usd`는 최소한 구현 컨텍스트와 검증 컨텍스트(앙상블·도구 호출 포함)의 LLM 토큰 비용 합산을 포함하며, 런타임의 단가표를 사용한다(MUST). 런타임은 자신의 계측 범위를 문서화해야 한다(MUST). `budget`은 회계 장치이지 보안 경계가 아니다 — egress 통제는 §9 참조.
- **시계 원점:** `max-duration`은 **해당 run의 시작 시각부터**의 경과 벽시계이며 일시정지(인간 게이트 대기 등)를 포함한다 — §5.2의 run 의미론과 일치하며, 모든 `stop`·`budget` 상한은 run당이다. run 시작 시각은 로그에서 확인 가능해야 한다(MUST).
- **`per-iteration`의 목적**은 총액 통제가 아니라 **iteration 간 비교 가능성**이다: 동일 예산 하의 점수라야 개선/정체 판정이 유효하다(§4.2.4). 선언 시 런타임은 각 iteration에 동일 상한을 강제해야 한다(MUST).
- **`per-iteration` 초과 시 부분 산출물 처리 (규칙 정교화 v0.4, 근거: 백로그 감사관 §4.2.6b):** 상한 도달로 iteration을 중단(kill)하면 구현 주체는 **불완전한 부분 산출물**을 작업트리에 남겼을 수 있다 — "산출물이 없다"고 단정하지 않는다. 런타임은 다음 두 정책 중 하나를 택하고 문서화해야 한다(MUST):
  - **(권장) 롤백:** 초과 iteration이 남긴 변경을 다음 iteration 시작 전 폐기해 이전 완결 상태로 복원한다. 이때 그 iteration은 (a) green-runs 리셋, (b) stagnation **미계상**(진행도 해시가 롤백으로 불변이므로), (c) max-iterations 계상.
  - **(허용) 잔존:** 부분 산출물을 작업트리에 남긴다. 이때 초과 iteration도 다음 iteration의 검증 대상이 되므로 (a) green-runs 리셋, (b) 그 후 첫 완주 iteration의 검증 결과를 진행도 해시로 stagnation에 **정상 계상**, (c) max-iterations 계상. 단 부분 산출물이 판정 자산을 오염시키지 않도록 §4.2.5 격리(검증 전 판정 자산 복원)는 유지되어야 한다(MUST).

#### 4.2.7 `log`

| 하위 필드 | 타입 | 기본값 |
|---|---|---|
| `path` | string | `state/logs/raw.jsonl` |
| `mode` | `append-only` | `append-only` (현재 유일한 값) |

- 모든 iteration의 시도·검증 결과·Δ(§4.2.8)는 로그에 기록되어야 한다(MUST).
- 로그는 append-only다: 어떤 프로세스도 기존 항목을 수정·삭제해서는 안 된다(MUST NOT).
- **원자적 쓰기:** 로그 쓰기는 단일 직렬화 지점을 통과해야 하며(MUST), 각 항목은 줄바꿈으로 종결된 하나의 원자적 append로 기록한다. iteration을 병렬화하는 런타임은 쓰기 직렬화를 보장해야 한다(MUST).
- 근거: append-only는 (a) 재작성 기반 루프의 context collapse 방지(ACE 실증), (b) objective hacking의 사후 탐지를 위한 변조-증거 감사 기록(DGM 실증), (c) 실패 이력 보존을 통한 인과 진단(Meta-Harness 실증)의 세 목적을 겸한다.

#### 4.2.8 로그 항목 스키마 (최소)

각 로그 항목은 최소한 다음을 포함해야 한다(MUST): `iteration`(int), `run-id`(활성화 세션 식별자, §5.2), `gene`(유전자 버전 식별자), `timestamp`(ISO 8601), `action`(수행 요약), `verify`(결과·점수), `delta`(방향 전환 시). 추가 필드는 자유다.

- **`gene`:** Evolution(§7)을 켜지 않은 루프는 고정값 `v0`(또는 LOOP.md 내용 해시)을 쓴다. Evolution 루프는 §7.1의 버전 식별자를 쓴다.
- **`delta`(Δ, 방향 전환):** 직전 iteration과 다른 접근을 택한 경우 — 폐기한 접근과 새 접근을 요약한 문자열. 방향을 유지한 iteration에는 생략한다.
- **경제/진단 계측 선택 필드 (v0.5, 근거: 필드 런 §6-8):** 로그 항목은 다음 선택 필드를 포함할 것을 권장한다(SHOULD): `tokens`(입력·출력·캐시 세분 권장, 구현·검증 컨텍스트 합산), `model`(사용 모델 식별자), `cost-usd`(해당 iteration 비용). 근거: 필드 런에서 러너 로그가 `timestamp`만 담아 런당 토큰·비용·모델을 사후 재구성할 수 없었다 — 경제성 분석("반복될수록 싸진다" 가설 §7.6의 검증)과 예산 감사의 입력이 로그에 없으면 확보되지 않는다. `budget`(§4.2.6)을 계측하는 Level 1+ 런타임은 이 값들을 이미 산출하므로 로그 기록은 추가 비용이 작다.
- **iteration 내부 Δ 기록 (SHOULD, 근거: 필드 런 §6-8):** 한 iteration 안에서 접근을 여러 번 전환한 경우, `delta`를 그 전환들의 목록/서사로 기록할 것을 권장한다(SHOULD) — iteration당 단일 Δ만 남기면 stagnation 판정(§4.2.4)의 입력이 빈약해지고 진단 해상도가 떨어진다. 방향 유지 iteration에는 여전히 생략한다.

#### 4.2.9 `skills`
이 루프가 소비하는 Agent Skills의 `name` 목록. 런타임이 Agent Skills를 지원하는 경우 해당 스킬을 통상 규격대로 로드 가능하게 해야 한다(SHOULD). 이 필드는 조건부 SHOULD이며, 특정 벤더 생태계에 대한 규범 의존이 아니다.

#### 4.2.10 `spec`
이 파일이 준수하는 명세 식별자(예: `agent-loops/0.5`). 선언을 권장한다(SHOULD). 런타임은 이 값으로 파일이 이 명세의 루프인지, 어떤 메이저 버전인지 판별한다 — 동명(同名)의 다른 포맷과 구별하는 지문이다.

## 5. 실행 모델

### 5.1 발견 (Discovery)
런타임은 설치된 루프들의 frontmatter 중 `name`·`description`·`goal`만을 카탈로그로 로드한다(SHOULD, 루프당 ~100토큰 수준). 본문·플레이북·로그는 기동 전 로드하지 않는다(SHOULD NOT).

### 5.2 기동 (Activation)
루프는 **명시적으로만** 기동된다: (a) 사용자 호출, (b) 스케줄러, (c) 상위 에이전트의 위임. 대화 중 모델이 임의로 루프를 기동해서는 안 된다(MUST NOT) — 루프는 비용·시간·부수효과가 큰 장기 실행물이다.

- **위임 계약:** 상위 에이전트 기동의 논리적 시그니처는 입력 = (루프 식별자, 작업 대상(§5.6), 활성화 파라미터), 반환 = §5.5의 종료 보고 구조체다(MUST). 구체 전송(CLI·API)은 런타임 자유다.
- **재활성화 의미론:** 이미 `state/`를 가진 루프의 재활성화는 새 **run**을 시작한다. run마다 iteration 카운터·green-runs 스트릭·`max-*` 상한은 리셋되며(MUST), 로그·플레이북·유전자 이력은 누적 유지된다. 각 로그 항목은 `run-id`를 갖는다(§4.2.8). — `stop`의 상한은 "한 run당" 상한이다. 상시 운영은 스케줄러가 매 주기 새 run으로 재기동하는 패턴으로 표현한다(무한 루프 선언 금지 §4.2.4와 양립).

### 5.3 반복 (Iteration)
1. 런타임은 구현 주체에게 **iteration 입력 최소 계약**을 전달한다(MUST): `goal`, LOOP.md 본문(변이 마커 포함), 직전 iteration의 `action`+`verify`+`delta` 요약, 현재 iteration 번호와 남은 예산. 그 이상의 로그·플레이북 제공은 선택이다(MAY).
2. 구현 주체가 `playbook/`을 조회하고(존재 시), 시도를 수행한다.
3. 산출물과 시도 요약이 로그에 기록된다(MUST).

각 iteration의 구현 컨텍스트는 신선하게 시작하는 것을 권장한다(SHOULD) — 기억은 대화 이력이 아니라 로그·플레이북(파일시스템)이 담당한다. 위 입력 계약이 유일한 기억 채널이므로 그 완전성이 중요하다.

### 5.4 검증 (Verification)
iteration마다 런타임이 `verify`(§4.2.5, 판정 자산 격리 포함)를 실행하고 결과를 로그에 기록한다(MUST). `stop.green-runs` 연속 통과 시 `completed`로 종료한다.

### 5.5 종료 (Termination)
런타임은 종료 시 §5.2 위임 계약의 반환값인 **종료 보고 구조체**를 산출해야 한다(MUST). 구체 전송(stdout·JSON·API 페이로드)은 런타임 자유이나, 필드명·타입은 상호운용을 위해 아래로 고정한다.

**종료 보고 기계 스키마 (v0.4, 근거: 백로그 감사관 §5.5):**

| 필드 | 타입 | 필수 | 의미 |
|---|---|---|---|
| `status` | enum | **MUST** | 종료 상태. §4.2.4의 6값 중 하나 (`completed`·`stalled`·`max-iterations`·`max-duration`·`budget-exceeded`·`aborted`) |
| `iterations` | int ≥ 0 | **MUST** | 이 run에서 실행한 iteration 수 |
| `run-id` | string | **MUST** | 이 run의 식별자(§5.2·§4.2.8). 로그 항목과 대조 가능해야 한다 |
| `gene` | string | **MUST** | 실행한 유전자 버전 식별자(§4.2.8) |
| `artifacts` | string | **MUST** | 산출물 위치(작업 대상 경로, §5.6) |
| `log` | string | **MUST** | 로그 경로(§4.2.7) |
| `summary` | string | SHOULD | "왜 성공/실패했는가" 자연어 요약 |

- 추가 필드는 자유다(예: `best-score`, `note`). 위 7필드는 최소 계약이다.
- 이 구조체는 로그에도 종료 항목으로 기록할 것을 권장한다(SHOULD) — 감사 시 run의 최종 상태를 로그만으로 재구성할 수 있다. (레퍼런스 러너 `runtime/looprun`이 이 형태를 산출한다: `{status, iterations, artifacts, log, run-id}` + extra.)

### 5.6 실행 컨텍스트 (작업 대상·cwd)
- **작업 대상(target):** 루프가 편집·검증하는 대상(코드베이스 등)은 활성화 시 파라미터로 지정한다(§5.2). 명시하지 않으면 기본값은 루프 디렉터리 자신이다.
- **verify 실행:** `verify.run`의 작업 디렉터리(cwd) 기본값은 작업 대상 루트다. 구현 주체가 만든 산출물과 `verify`는 같은 작업트리를 공유한다(단, 판정 자산은 §4.2.5에 따라 격리된 사본에서 실행). 환경변수 주입 규칙은 런타임이 문서화해야 한다(MUST).

### 5.7 크로스-세션 연속성 아티팩트 (선택, v0.5)

> 근거: 필드 런 §6-5. 여러 run이 인간의 재점화로 이어지는 실전 패턴(§5.2 재활성화)에서, `state/`의 세 채널만으로는 "지금 어디까지 왔고 다음에 무엇을"의 **재개 서사**가 재구성되지 않았다 — 로그는 얇은 원장, 유전자는 diff, 플레이북은 노이즈다. 필드 런에서 오퍼레이터가 이 요약을 손으로 발명(`HANDOFF.md` 자생)한 것이 공백의 증거다.

런타임 또는 오퍼레이터는 크로스-세션 인수인계 요약을 **선택 산출물**로 유지할 수 있다(MAY). 표준화하는 것은 위치와 최소 내용뿐이다:

- **위치:** `state/HANDOFF.md`(§2). 배포 금지(§8.1), 자유형 마크다운.
- **최소 내용 (있을 경우 SHOULD):** (a) 최근 run들의 결과 요약(종료 상태·달성/미달 목표), (b) 현재 미해결 항목·다음 착수점, (c) 마지막 유전자 버전과 그 근거, (d) 인간 개입이 필요한 열린 결정.
- **판정 격리:** 이 아티팩트는 자유형 서사이므로 어떤 `verify`에도 입력되어서는 안 된다(MUST NOT) — 자기보고 금지(§1.2 원칙 2)를 우회하는 채널이 되지 않도록. 오퍼레이터·후속 세션의 오리엔테이션 전용이다.
- 이 요약은 §5.5 종료 보고 구조체의 `summary` 필드를 run 경계 너머로 누적한 것으로 볼 수 있으나, 종료 보고가 기계 스키마인 데 반해 이것은 인간용 서사다. 둘은 상보적이다.

## 6. 준수 수준 (Conformance Levels)

| 수준 | 이름 | 요구사항 |
|---|---|---|
| **Level 0** | Document | 파일 형식 준수(§3–§4)만. 런타임 행위 요구 없음 — LOOP.md는 에이전트가 세션 안에서 읽고 따르는 문서로 기능한다. 규범적 강제(자기보고 금지·정지·로그)는 Level 1+에서 발생하며, Level 0은 상위 수준으로의 무수정 이행을 보장하는 진입점이다 |
| **Level 1** | Runner | §5 실행 모델 전체 + `stop`·`budget` 강제 + append-only 로그 + 검증 분리·판정 자산 격리(§4.2.5) |
| **Level 1.5** | Refine | Level 1 + 플레이북 정제·승격 사다리(§7.5, dedup 게이트 포함, 단 **규칙(rule)까지만** — 공식 승격은 불가) + 변이 표면 강제(§7.3, 압축 무손실 목록화 포함) + 인간 게이트(§7.4) + 메타레이어 강화 이력(§7.7). 홀드아웃(§7.2)은 **인간 게이트로 대체 가능** |
| **Level 2** | Evolution | Level 1.5 + 홀드아웃 접근 통제(§7.2, 기술적 차단) + 공식 승격(§7.5)·래칫·재검증 전체 + §8 배포 규칙 |

- **Level 1.5 "Refine"의 취지 (v0.4, 근거: 백로그 대형 8a):** 실행 이력에서 학습해 플레이북을 정제하고(§7.5) 선언된 변이 표면(§7.3)을 인간 게이트(§7.4) 아래 갱신하되, **결정론 실행물(공식) 자동 승격과 홀드아웃 기반 자동 채택은 아직 켜지 않은** 중간 준수 수준이다. 홀드아웃 격리(§7.2)의 기술적 방벽 대신 **모든 변이·규칙 승격을 인간 게이트로 통과**시켜 안전을 확보한다 — 즉 §7.5의 승격 사다리에서 지위는 **규칙(rule)까지만** 오를 수 있고, 공식(formula)으로의 승격은 Level 2에서만 허용된다(MUST). Level 1.5는 §7.2의 자동 홀드아웃 리허설 인프라 없이 진화의 학습 이득을 얻으려는 팀을 위한 것이다.
- 런타임은 자신이 준수하는 수준과 계측 범위(§4.2.6)를 문서화해야 한다(MUST).
- **기계 검증 가능/불가 구분:** 로그 스키마 준수(§4.2.8), `stop` 상한 강제, append-only 무변조는 외부에서 기계 검증 가능한 MUST다. 검증 분리·판정 자산 격리·자기보고 금지는 런타임 내부 속성이라 기계 검증이 어렵다 — §6.1의 적합성 픽스처가 이 간극을 좁힌다.

### 6.1 적합성 픽스처 (개요, v0.4)

> 근거: 백로그 대형 8b. 이 절은 **정본 적합성 스위트의 개요**다 — 픽스처 루프의 목록과 각 픽스처가 관측해야 할 기대 관측치의 윤곽만 정의한다. 전체 실행 가능 스위트(픽스처 자산·자동 채점 하네스)는 **v0.6 과제**다.

정본 픽스처는 러너가 관측 가능한 행위로 준수를 증명하도록 설계한다. 각 픽스처는 (입력 루프, 작업 대상, 기대 관측치) 3항으로 기술한다.

| 픽스처 | 검사하는 MUST | 기대 관측치 (개요) |
|---|---|---|
| `fx-green-basic` | §5 반복·검증·완료 | 버그 심은 대상을 유한 iteration에 수리, 종료 `status: completed`, green-runs 충족 |
| `fx-stop-max-iter` | §4.2.4 정지 상한 | 통과 불가 대상에서 `max-iterations`로 정확히 상한 회에 종료 |
| `fx-asset-tamper` | §4.2.5 판정 자산 격리 | 구현 주체가 테스트를 약화시켜도 검증 전 복원되어 통과 위조 실패, 변조가 로그에 기록 |
| `fx-asset-undeclared` | §4.2.5 격리 거부 규칙 | `assets`·clean-ref 모두 없으면 **실행 거부**(비-Level 1 표기 불가) |
| `fx-append-only` | §4.2.7 로그 불변 | 기존 로그 항목 무수정·무삭제, 원자적 append, run 간 누적 |
| `fx-run-semantics` | §5.2 재활성화 | 재활성화 시 카운터 리셋·`run-id` 신규 부여, 로그·플레이북 누적 |
| `fx-score-stagnation` | §4.2.4 무진전 | 개선 없는 대상에서 `stalled` 종료, 진행도 해시 기준 관측 |
| `fx-agent-verdict` | §4.2.5 판정 프로토콜 | `agent` 검증기의 `VERDICT:`/종료코드 판독, `n>1` 합의 집계 |

- 러너는 지원 기능에 해당하는 픽스처를 통과해야 준수를 주장할 수 있다(SHOULD, 전체 규범화는 v0.6). 미지원 기능은 침묵이 아니라 **실행 거부**로 처리해 픽스처가 이를 관측할 수 있어야 한다(MUST) — 레퍼런스 러너 `runtime/looprun`이 이 거부 원칙(assets·agent·cost 미지원 시 거부)의 실측 사례다.

## 7. Evolution 확장 (선택)

> 이 절 전체는 선택적 확장이다. Core(§1–6)만으로 완결적 표준이며, Evolution은 "반복 사용되는 루프가 실행 이력에서 학습해 스스로 갱신되는" 시스템을 원하는 경우 켠다.
> **적용 수준 (v0.4):** 이 절의 요구사항은 **Level 1.5(Refine)와 Level 2(Evolution)**에 적용된다. Level 1.5는 플레이북 정제(§7.5, 규칙까지)·변이 표면(§7.3)·인간 게이트(§7.4)를 강제하되 홀드아웃(§7.2)을 인간 게이트로 대체하고 공식 자동 승격을 금한다. Level 2는 §7 전체(홀드아웃 기술 차단·공식 승격·래칫·재검증)를 강제한다. 각 하위 절은 어느 수준부터 적용되는지 표시한다.

### 7.1 진화 루프 개요

```
state/logs (v(N)의 이력) → 정제 → 개선 에이전트 → LOOP.md 변이 제안
   → [자동 적용 | 래칫 | 인간 게이트] → 홀드아웃 리허설 (신구 동일 잣대) → 유전자 v(N+1) 커밋
```

- 개선 에이전트는 구현 에이전트와 분리된 컨텍스트여야 한다(MUST).
- **정제 실행 시점 (v0.5, 근거: 필드 런 §6-6):** Level 1.5+ 런타임은 run 종료 후 정제(개선 에이전트 실행)를 **기본 절차로 수행할 것을 권장한다(SHOULD)** — 필드 런에서 사용자가 "유전자를 남겨 진화시켜라"를 두 번 상기시켜야 evolve가 돌았다. 정제가 옵트인이면 학습 이득이 인간의 기억에 의존해 소실된다. 런타임은 이 기본 동작을 선택 필드 `evolve`로 제어할 수 있게 한다(MAY): `evolve: auto`(run 후 자동 정제, 권장 기본값) · `evolve: manual`(명시 호출 시에만) · `evolve: off`(정제 비활성). 미선언 시 런타임 기본은 `auto`를 권장한다(SHOULD). 어느 경우든 정제 산물의 채택은 준수 수준별 게이트(§7.4·§7.5)를 그대로 거친다 — 자동 실행은 "정제를 돌린다"이지 "변이를 무검증 채택한다"가 아니다.
- 승인된 변이는 새 **유전자 버전**이 된다: git 커밋 또는 `state/genes/` 스냅샷으로 기록하고(MUST), 이후 로그 항목은 해당 유전자 버전을 태깅한다(MUST, §4.2.8).
- **계보:** 유전자 계보는 선형이다 — 새 유전자의 부모는 항상 직전 채택 버전이다. 유전자 기록은 선택적 `parent` 필드(부모 버전 식별자)를 포함할 것을 권장한다(SHOULD). 선형 모드에서 `parent`는 자명하지만, 이를 기록해 두면 미래 버전에서 계보 분기(트리 탐색)를 하위 호환으로 도입할 수 있다. **분기 탐색 자체는 현재 명시적 비목표(non-goal)다** — 실패 이력 보존은 append-only 로그(§4.2.7)가 이미 보장하므로 분기 부재가 인과 진단을 제한하지 않는다. (결정 이력: `wiki/contradictions/001-greedy-vs-archive.md`)

### 7.2 `holdout` — 홀드아웃 격리

| 하위 필드 | 타입 | 의미 |
|---|---|---|
| `path` | string | 홀드아웃 자산 경로 (권장: `state/holdout/`) |
| `improver-access` | `none` | 현재 유일한 값. 개선 에이전트의 읽기 금지 |
| `refresh` | `manual` \| `on-model-change` \| `per-generation` | 홀드아웃 갱신 정책 |

- 런타임은 개선 에이전트의 홀드아웃 접근을 **기술적으로**(파일시스템 권한·샌드박스) 차단해야 한다(MUST). 프롬프트 지시만으로는 준수가 아니다.
- **로그 유출 차단 (v0.3):** 홀드아웃 검증의 결과는 로그에 pass/fail과 집계 점수만 남긴다(MUST). 항목별 상세(입력·기대값·실제값 등)를 로그에 기록해서는 안 된다(MUST NOT). — 근거: 개선 에이전트는 §7.1에 따라 로그를 정당하게 읽으므로, 홀드아웃 상세가 로그에 남으면 파일 접근 차단을 우회해 정제만으로 홀드아웃을 재구성할 수 있다. 방벽은 파일뿐 아니라 로그 채널에도 있어야 한다.
- 유전자 변이의 채택 판정은 신·구 유전자를 **동일한 홀드아웃, 동일한 잣대**로 비교한 결과에 근거해야 한다(MUST).
- 기반 모델 변경 시 재검증을 권장한다(SHOULD): `refresh: on-model-change`.
- 근거: (a) DGM은 평가 함수를 에이전트로부터 은닉했고 "은닉하지 않으면 objective hacking이 더 잦다"를 실증했다. (b) 은닉만으로는 불충분하다 — DGM node 114는 은닉 하에서도 검출을 우회했다(→ 게이트 §7.4가 함께 필요). (c) 정적 프록시의 무한 최적화는 진짜 성능을 상승 후 하락시킨다(Goodhart, ICML 2023); 평가 신호 갱신이 이를 완화한다.

### 7.3 `mutable` — 변이 표면 선언

| 하위 필드 | 타입 | 의미 |
|---|---|---|
| `free` | string[] | 개선 에이전트가 자유 수정 가능한 대상 |
| `ratchet` | object[] | **강화 방향으로만** 자동 수정 가능한 대상 (방향 명시 필수) |
| `gated` | string[] | 수정 제안 시 인간 승인 필수인 대상 |

- **대상 지정 문법:** frontmatter 대상은 점 표기 키 경로(예: `verify[2].consensus`, `stop.stagnation`), 본문 대상은 마커 이름으로 지정한다. 세 목록 어디에도 없는 frontmatter 필드와 마커 밖 본문은 **불변**이다(MUST). `goal`·`verify`·`holdout`·`log`는 `free`나 `ratchet`에 넣어서는 안 된다(MUST NOT).
- **본문 변이 표면:** `<!-- LOOP:FREE -->` … `<!-- /LOOP:FREE -->` 블록 안만 개선 에이전트가 재작성할 수 있다(MUST). 마커 밖 본문은 불변이다. (선례: AlphaEvolve `# EVOLVE-BLOCK`.) FREE 블록의 프로즈는 `verify`·`stop`·`gated` 항목의 무시·우회를 지시할 수 없다(MUST NOT) — 게이트·정제 주입 검토(§7.4, §9)는 FREE 블록 diff에서 이런 무력화 지시를 검사한다.
- **압축 무손실 보증 (v0.5, 근거: 필드 런 §6-7):** 개선 에이전트가 FREE 블록을 **재작성(압축·정리)** 할 때, 재작성으로 **제거되는 기존 지시**를 명시적으로 목록화해야 한다(MUST) — 재작성 제안에 "제거된 지시 목록(각 항목: 원문 + 제거 사유)"을 첨부하지 않은 FREE 변이는 채택해서는 안 된다(MUST NOT). 근거: 필드 런에서 유전자 압축 중 "ui-criteria를 읽으라"는 지시가 소리 없이 탈락해 후속 런의 판정 기준 참조가 끊겼다. 재작성은 프로즈를 줄이지만 지시의 **무언(無言) 탈락**은 능력의 회귀이며, 이는 인간 게이트(§7.4)나 리뷰어가 관측할 수 있어야 한다. (순수 표현 변경으로 지시 집합에 변화가 없으면 목록은 비어 있다 — 빈 목록의 명시도 "탈락 없음"의 확인으로서 유효하다.)
- **래칫 의미론:** `ratchet` 항목은 강화 방향을 명시해야 한다(예: `{ pass-threshold: higher-is-stricter }`). 런타임은 완화 방향 자동 변경을 거부해야 한다(MUST); 완화는 `gated` 경로로만 가능하다. **임계값이 래칫되는 지표의 조작적 정의**(`score`의 명령·입력집합)는 함께 동결되어 gated여야 한다(MUST) — 임계값만 올리고 측정 대상을 쉬운 프록시로 바꾸는 우회(명목상 강화, 실효상 완화)를 막기 위함이다. (선례: Notion 프로덕션 ESLint 래칫.)

### 7.4 인간 게이트

- `gated` 항목의 변이, 래칫 항목의 완화, 루프 구조 변경 제안은 인간 승인 전까지 적용해서는 안 된다(MUST NOT).
- **승인 요청 첨부(MUST):** 변이 diff, 제안 근거, 홀드아웃 신구 비교 결과. **여기서 "변이 diff"는 직전 게이트 이후 발생한 패키지 전체 delta**를 뜻한다 — 유전자 변경뿐 아니라 게이트-프리 경로로 `scripts/`·`playbook/generic/`에 착지한 규칙·공식 승격 목록을 포함한다(MUST). (유전자 diff만 제시하면 실제 능력 변화가 게이트를 우회한다.)
- 승인 대기 중에도 루프는 현행 유전자로 계속 운행할 수 있다(MAY) — 진화의 정체가 운영의 정지가 되어서는 안 된다.

### 7.5 플레이북과 승격 사다리

정제된 지식은 3단계 지위를 갖는다:

| 지위 | 진입 조건 | 실행 방식 | 적용 수준 |
|---|---|---|---|
| 휴리스틱 | 정제에서 후보로 식별 | LLM이 참고 자료로 활용 | Level 1.5+ |
| 규칙 | k회 에피소드에서 유효 (k는 루프 정의) | LLM이 우선 적용 | Level 1.5+ |
| 공식 | 홀드아웃 통과 + 승격 검토(아래) | `scripts/`의 결정론 실행물로 컴파일, 라우터가 전제조건 충족 시 LLM 없이 실행 | **Level 2만** |

- **수준별 상한 (v0.4, 근거: 백로그 대형 8a):** Level 1.5(Refine)에서 지위는 **규칙까지만** 승격할 수 있다(MUST) — 공식(결정론 실행물)으로의 승격은 홀드아웃 리허설(§7.2)과 신규 능력 검사를 요구하므로 Level 2에서만 허용된다. Level 1.5는 규칙 승격을 홀드아웃 대신 인간 게이트(§7.4)로 통과시킨다.

- **플레이북 항목 최소 스키마(MUST):** 각 항목은 `id`, `status`(heuristic·rule·formula), `evidence`(진입 근거 로그 참조), `episodes`(유효 카운트 k), `preconditions`(적용 전제)를 갖는다. `playbook/generic/`(배포 대상)은 이 스키마를 준수해야 이식 가능하다.
- **canonical dedup 게이트 (v0.5, 근거: 필드 런 §6-2):** 정제가 신규 플레이북 항목을 기록하기 전, 기존 항목과 **의미 기반으로 대조·병합해야 한다(MUST)** — 신규 항목이 기존 항목과 같은 인사이트면 새 `id`를 만들지 말고 기존 항목의 `episodes`를 누적하고 `evidence`를 추가한다. 근거: 필드 런에서 플레이북 24건이 실질 ~5개 인사이트였고, dedup 부재가 승격을 **양방향으로 왜곡**했다 — 한 인사이트가 5개 id로 파편화돼 각각 `episodes=1`에 고착(승격 임계 k=2를 못 넘김)되는 과소 계상과, 같은 3개 run이 3개 항목에 각각 `episodes=3`으로 계상되는 지지도 인플레이션이 동시에 발생했다. 대조는 표면 문자열이 아니라 의미로 해야 한다(단·복수만 다른 항목도 병합 대상). 병합 판정 자체는 정제(개선 에이전트)가 수행하며, 병합 이력은 로그·`evidence`에 남긴다(SHOULD). **`episodes` 계상 규칙:** 한 인사이트의 `episodes`는 그것을 **독립적으로 지지하는 서로 다른 run/에피소드의 수**여야 한다(MUST) — 같은 근거를 여러 항목에 중복 계상해서는 안 된다(MUST NOT).
- **공식 승격 검토 (v0.3, 중요):** 규칙 → 공식 승격(즉 `scripts/`의 LLM-없는 실행물이 되는 것)은 홀드아웃 통과만으로 자동 진행해서는 안 된다(MUST NOT). 승격 시 (a) 내용 검토(§9)와 (b) 신규 능력 diff(네트워크 egress·작업 대상 밖 파일 쓰기 등)를 검사해야 하며, 신규 부수효과가 있으면 인간 게이트(§7.4)를 거쳐야 한다(MUST). — 근거: 검토 없는 자동 승격은 외부 콘텐츠의 주입 지시가 결정론 실행물로 굳어지는 자기 공급망 공격 경로다(§9.2).
- **강등:** 공식·규칙이 검증 실패를 유발하면 런타임은 즉시 한 단계 강등하고 사유를 로그에 남겨야 한다(MUST).
- **격리:** 인스턴스 특이 지식은 `playbook/local/`에 두어야 하며(MUST), 다른 인스턴스 실행 시 로드되어서는 안 된다(MUST NOT).
- 근거: 발견된 전략의 결정론 재실행이 정확도 10%→99%를 만든 실증(Dynamic Cheatsheet).

### 7.6 경제 텔레메트리 (권장)

Level 2 런타임은 로그로부터 런별 경제 지표 — iteration 수, 비용, **결정론 실행 비율**(라우터가 공식으로 처리한 비중) — 를 산출·표시할 것을 권장한다(SHOULD). 이 산출의 입력은 §4.2.8의 경제 계측 선택 필드(`tokens`·`model`·`cost-usd`)다 — 이 필드들이 로그에 없으면 경제 지표를 사후 재구성할 수 없다(필드 런 §6-8의 실측 공백). "반복될수록 싸진다"는 이 표준의 **가설**이며(공개 실증은 아직 없다), 각 인스턴스가 자기 데이터로 이를 증명하거나 반증하게 하는 것이 표준의 신뢰성에 기여한다.

### 7.7 Anti-Goodhart 메타레이어 강화 (v0.5, 근거: 필드 런 §3·§4)

> 이 절은 **프록시 타당성(축 B, §4.2.5a)의 런 간 동학**을 다룬다. §4.2.5a가 "프록시를 처음부터 목표에 가깝게 설계"라면, 이 절은 "런을 거치며 프록시(판정 자산·기준)를 점점 속이기 어렵게 강화"다. **적용 수준:** Level 1.5+(강화 절차·이력)·Level 2(자동 래칫 결합).

첫 실전 필드 런에서 검증 기준(`ui-criteria`·테스트 자산)이 런 사이 10줄→62줄로 **단조 강화**됐다 — 이는 루프 내부가 아니라 인간의 스크린샷 비평 → 오퍼레이터 승격이라는 **수동 사이클**의 산물이었다. 판정기를 시간이 갈수록 속이기 어렵게 만드는 이 핵심 노동이 스펙 밖에 있었다. 이 명세는 이를 두 레이어로 분리해 1급 개념으로 명문화한다:

- **레이어 1 — 런 내 불변 (기존, §4.2.5):** 한 run 안에서 판정 자산은 구현 주체가 변경할 수 없다(격리). 이는 축 A(실행 독립성)의 보장이며 이미 규범이다.
- **레이어 2 — 런 간 메타레이어 강화 (신규):** run 경계를 넘어 판정 자산·기준 자체를 **강화 방향으로만** 갱신하는 별도 레이어다. 이 갱신은 구현 주체가 아니라 **인간 또는 개선 에이전트**가, **인간 승인**(Level 1.5의 게이트 §7.4, 또는 Level 2의 래칫 §7.3) 아래에서 수행한다.

레이어 2에 대한 규범:

- **단조성 (MUST):** 메타레이어 강화는 판정을 **더 엄격하게** 만드는 방향이어야 한다(기준 추가·임계 상향·우회 차단). 완화(기준 삭제·임계 하향)는 강화가 아니며, `gated` 경로(§7.4)로만 가능하다 — 이는 `ratchet` 의미론(§7.3)의 판정 자산 버전으로의 확장이다.
- **강화 이력의 구조화 (SHOULD):** 각 강화 이벤트를 `state/ratchet/`(§2)에 구조화 기록할 것을 권장한다(SHOULD). 최소 필드: `who`(강화 주체 — 인간/개선 에이전트), `why`(근거), `from-critique`(어느 비평·관측에서 도출됐는지, 예: 로그·핸드오프 참조), `target`(강화된 판정 자산 경로), `direction`(무엇을 더 엄격하게 했는지), `timestamp`. 근거: 필드 런에서 "판정을 강화하는 노동"의 소유자·절차·계보가 어디에도 남지 않아 재구성이 불가능했다. 이 이력은 §7.4 게이트의 "변이 diff = 패키지 전체" 요구와 결합해, 능력 변화가 게이트를 우회하지 못하게 한다.
- **자기보고 금지와의 정합 (MUST NOT):** 메타레이어 강화의 **입력**이 되는 비평은 구현 주체 자신의 자평이어서는 안 된다(MUST NOT) — 인간 비평·독립 관측·홀드아웃 결과여야 한다. 구현 주체가 자기 판정 기준을 스스로 강화하면 원칙 2(자기보고 금지)가 형해화된다.
- **필드 런 사례:** run2(green·무동작) → 사이에 `ui.spec.ts`가 강화 → run3(동일 유전자가 pre-check fail). "통과하던 앱을 다시 실패하게 만드는" 이 전이가 레이어 2가 실제로 작동한 관측 증거다 — 강화 전 확인(무동작 런)은 낭비가 아니라 래칫의 정상 단계다.

이 절은 축 B를 **완전히 자동화하지 않는다** — 무엇이 "목표를 더 잘 담보하는 강화인가"의 판단은 여전히 인간(또는 홀드아웃)에 의존한다. 스펙이 표준화하는 것은 그 강화가 **일어날 자리·방향 제약·감사 가능한 이력**이다.

## 8. 배포 (Distribution)

### 8.1 패키징

배포 패키지는 다음을 포함한다: `LOOP.md`, `scripts/`, `playbook/generic/`, (선택) 라이선스·문서.

다음은 포함해서는 안 된다(MUST NOT): `state/**`(로그·유전자 스냅샷·홀드아웃), `playbook/local/**`.

> 홀드아웃 자산을 배포하면 모든 수신자의 개선 에이전트에 대해 홀드아웃이 사전 노출된 것과 같다. 홀드아웃은 인스턴스가 각자 구성한다.

- **프로비넌스 (v0.3):** 마켓플레이스에 게시되는 루프의 `scripts/`는 서명·출처(provenance) 정보를 동반해야 한다(MUST). 미서명·미검증 출처의 루프는 첫 실행 시 샌드박스에서 실행해야 하며(MUST, §9와 연동), 요구 능력(egress·fs 범위)을 `compatibility`에 선언해야 한다(MUST). — 근거: `scripts/`의 공식은 설치 즉시 인스턴스 권한으로 LLM 없이 실행되므로(§7.5), 검토 없는 배포는 공급망 공격 벡터다.

### 8.2 마켓플레이스 메타데이터

카탈로그 노출에는 frontmatter의 `name`·`description`·`goal`·`verify`(방식 요약)·`license`·`compatibility`를 사용한다(SHOULD). `verify`의 공개는 "이 루프가 스스로를 어떻게 채점하는가"라는 신뢰 신호다.

- 배포자는 집계 지표(누적 런 수·홀드아웃 통과 이력)를 동봉할 수 있으나(MAY), 원시 로그가 §8.1로 배제되어 이는 자기보고에 불과하다. 이런 지표를 신뢰 신호로 제시하려면 검증 가능한 증거(서명된 로그 다이제스트, 제3자 attestation)를 동반해야 한다(MUST). 그렇지 않으면 신뢰 신호로 표기해서는 안 된다.

### 8.3 스킬로 내보내기

승격 사다리의 "공식"은 Agent Skills의 `scripts/`와 동형이므로, 루프는 자신의 공식들을 SKILL.md 패키지로 내보낼 수 있다(MAY). 내보낸 스킬의 `name`·`description`은 루프 frontmatter에서 파생하되 Agent Skills 명세를 준수해야 한다(MUST).

### 8.4 서명·attestation (자리 확보, v0.4 — 비규범)

> 근거: 백로그 대형 8c. 이 절은 **요구사항의 자리와 목표만** 정의한다. 구체 방식(서명 알고리즘·attestation 포맷·검증 체인)은 **비규범**이며 v0.6 이후 확정한다.

§8.1의 프로비넌스와 §8.2의 검증 가능 신뢰 신호는 **암호학적 서명과 attestation**을 요구 지점으로 둔다. 향후 버전이 다음을 규범화할 자리를 예약한다:

- **패키지 서명:** 배포되는 `scripts/`·`LOOP.md`의 무결성·출처를 검증 가능하게 하는 서명. 미서명 패키지의 취급은 §8.1(첫 실행 샌드박스)을 따른다.
- **로그 다이제스트 attestation:** §8.2가 요구하는 "검증 가능한 증거"의 형식 — 서명된 로그 다이제스트 또는 제3자 attestation으로 집계 지표(누적 런·홀드아웃 통과)를 뒷받침한다.
- **유전자 계보 서명:** §7.1 유전자 버전 커밋에 서명해 진화 이력의 변조를 탐지(append-only 로그 §4.2.7의 감사 목적과 결합).

이 명세는 특정 서명 방식(Sigstore·minisign·GPG 등)을 규정하지 않는다 — 런타임·마켓플레이스가 방식을 택하고 문서화한다. 서명 부재가 Level 1/1.5/2 준수를 막지는 않으나(로컬 실행 시나리오), 마켓플레이스 배포의 신뢰 신호로 제시하려면 이 자리의 요구를 충족해야 한다(MUST, §8.2).

## 9. 보안 고려사항

1. **스크립트 실행:** `scripts/`·`verify.run`은 임의 코드 실행이다. 런타임은 신뢰되지 않은 루프의 코드를 샌드박스에서 실행해야 한다(SHOULD; 마켓플레이스 출처는 §8.1에 따라 MUST). 위험 모델은 Agent Skills 번들 스크립트와 동일하다.
2. **정제 주입:** 플레이북은 실행 이력에서 자동 생성되므로, 처리한 외부 콘텐츠의 지시문이 플레이북으로 승격되는 주입 경로가 존재한다. 정제 산물은 데이터로 취급하고, 규칙 승격 시 내용 검토를 권장하며(SHOULD), 공식 승격(결정론 실행물화) 시 내용 검토와 신규 능력 검사는 필수다(MUST, §7.5).
3. **홀드아웃 무결성:** §7.2의 기술적 차단(파일 + 로그 채널)이 없다면 Evolution 확장을 준수한다고 표시해서는 안 된다(MUST NOT).
4. **판정 자산 무결성:** §4.2.5의 판정 자산 격리(`assets` 선언 또는 clean-ref 체크아웃)가 없으면 Level 1을 준수한다고 표시해서는 안 된다(MUST NOT). 어느 격리 수단도 확보되지 않은 루프는 실행을 거부해야 한다(MUST, §4.2.5). 판정 자산과 가변 소스가 같은 파일에 공존하면 파일 단위 격리로는 불완전하므로 §4.2.5의 저작 지침(파일 분리 SHOULD)을 따른다. — 격리(축 A)가 완전해도 **프록시가 목표를 배신**(축 B)하면 판정은 무의미하다: 페널티-only 산식·정확문자열 blocklist·taste 자기보고는 게이밍 표면이다. 축 B 완화는 §4.2.5a(설계)와 §7.7(런 간 강화)이 다룬다.
5. **예산 vs 보안:** `budget`은 런타임 매개 호출만 계측하는 회계 장치다. 구현 주체가 자체 키로 외부 API를 호출하거나 서브프로세스로 연산을 오프로드하면 계측을 우회할 수 있다. 예산을 안전·공정성 경계로 삼으려면 egress 통제 샌드박스가 필요하다(SHOULD). `budget` 없는 무인 스케줄 실행은 권장하지 않는다(SHOULD NOT).

## 부록 A. 최소 예시 (Core만 — 필수 5필드)

```markdown
---
name: fix-regressions
description: >
  Retries fixes for failing regression tests until the suite passes.
  Use nightly or after large merges.
goal: All regression tests pass on main
stop: { max-iterations: 15, stagnation: 3 }
verify: { run: "npm test" }
---

# 운영 지침
1. 로그에서 직전 시도의 Δ를 확인한다.
2. 실패 테스트를 하나 골라 원인을 고친다 — 한 iteration에 한 단위 작업만.
3. 판정은 verify가 한다. 완료 선언을 하지 말 것.
```

> 이 최소 예시는 유효한 파일(Level 0)이다. **Level 1 실행** 시 판정 자산 격리(§4.2.5)가 필요하므로 러너는 (a) 작업 대상이 git 등 clean-ref를 제공해 검증을 클린 커밋에서 체크아웃하거나, (b) 아래처럼 `verify.assets`를 선언해야 한다 — 둘 다 없으면 러너는 실행을 거부한다. 예: `verify: { run: "npm test", assets: ["test/", "package.json"] }`.

로그 항목 예시(§4.2.8):
```json
{"iteration": 4, "run-id": "2026-07-06T02:00Z", "gene": "v0", "timestamp": "2026-07-06T02:11:03Z",
 "action": "flaky auth test의 race condition에 await 추가", "verify": {"pass": false, "failed": 2},
 "delta": "이전엔 테스트를 skip 처리 시도 → 근본 원인(race) 수정으로 전환"}
```
