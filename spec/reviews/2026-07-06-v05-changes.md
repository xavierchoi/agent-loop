# 스펙 v0.4 → v0.5 개정 결정 로그 — 2026-07-06

## 개요

첫 실전 필드 런(`research/field-runs/2026-07-06-moneta-live-ui.md`) §6의 v0.5 백로그 8건 + v0.4 개정에서 파생된 열린 질문 5건(`2026-07-06-v04-changes.md`)을 반영해 v0.4 → v0.5 Draft로 개정.

**핵심 서사 — 축 B의 스펙 진입.** v0.4까지 검증 규범은 **축 A(실행 독립성: 자기보고 금지·판정 자산 격리)**만 다뤘다. 필드 런은 "판정이 완전히 독립적(자산 변조 0)이면서도 **측정 지표가 목표를 배신하는 프록시**"(시각 점수 만점 = 무채색 빈 화면)를 실측했다. 이를 **축 B(프록시 타당성)**로 명명하고, 설계 시점(§4.2.5a)과 런 간 강화(§7.7) 양쪽에서 스펙 언어로 도입한 것이 이번 개정의 축이다.

**제약 준수:** Core 미니멀리즘 유지 — 필수 5필드 불변, 신규 **필수** 필드 0건. 신규 필드는 전부 선택(`evolve`, 로그 `tokens`/`model`/`cost-usd`)이며 기존 객체의 하위 필드로 additive. "증류" 미사용, "정제(refinement)"로 통일. `wiki/`·`runtime/`·`research/` 무수정 — `spec/`만 편집.

## 새로 생긴 열린 질문 (v0.6+ 후보)

1. **축 B의 픽스처화.** §4.2.5a(페널티-only 최적점 검사·정량 게이트)와 §7.7(단조 강화)은 현재 SHOULD 권고라 기계 관측이 어렵다. "자명한 열화 산출물 → 만점이면 프록시 배신"류의 관측 픽스처를 v0.6 적합성 스위트에 넣을 수 있는지 미결.
2. **`evolve: auto`와 예산의 상호작용.** run 후 자동 정제가 도는 경우 정제 자체의 토큰·비용이 `budget`(§4.2.6) 계측에 어떻게 잡히는지(run 예산 안? 별도?) 미규정.
3. **강화 이력의 배포 정책.** `state/ratchet/`는 §8.1로 배포 금지(state 하위)이나, 판정 기준 강화 계보는 "왜 이 루프가 이 기준을 쓰는가"의 신뢰 신호일 수 있다 — 유전자 계보처럼 요약본을 배포 가능하게 할지 미결.
4. **HANDOFF 자동 생성 주체.** §5.7은 위치·내용만 표준화하고 생성 주체(런타임 자동 vs 오퍼레이터 수동)를 열어뒀다. 자동 생성 시 어느 컨텍스트가 쓰는지, 그것이 자기보고 채널이 되지 않는지 v0.6 검토.
5. **dedup 병합 판정의 감사.** canonical dedup(§7.5)의 "의미 기반 병합"을 개선 에이전트가 수행하는데, 잘못된 병합(서로 다른 인사이트를 합침)의 탐지·되돌림 절차가 미규정.
