# 스펙 v0.3.1 → v0.4 개정 결정 로그 — 2026-07-06

## 개요

v0.4 백로그 8건(`2026-07-06-v02-critique.md`의 "v0.4 백로그(통합)")을 반영해 v0.3.1 → v0.4 Draft로 개정. 백로그는 v0.3 반영 직후 Level 1 레퍼런스 러너(`runtime/looprun`) 구현 시험 + 독립 감사에서 도출된 것으로, 스펙 문구가 이 구현의 실측 관례(assets 거부 원칙, agent 검증기 거부, stagnation 다이제스트)와 정합하도록 맞췄다.

**제약 준수:** Core 미니멀리즘 유지 — 필수 5필드 불변, 신규 **필수** 필드 0건. 신규 필드는 전부 선택(`verify.assets`, `stop.progress-paths`)이며 기존 객체의 하위 필드로 additive. "증류" 미사용, "정제(refinement)"로 통일. `runtime/**` 무수정.

## 새로 생긴 열린 질문 (v0.5+ 후보)

1. **파일 미만 격리 메커니즘.** SHOULD 저작 지침으로 우회했으나, 테스트·소스 공존이 불가피한 언어/프레임워크(예: Rust `#[cfg(test)]` 인라인, doctest)에서는 파일 분리가 부자연스럽다. AST/라인 범위 잠금 또는 "판정 자산 = 별도 clean-ref만" 강제 중 택할지 미정.
2. **agent 검증기 판정 형식 우선순위.** `VERDICT:` 줄과 종료코드가 **모순**될 때(예: 마지막 줄 `VERDICT: pass`인데 종료코드 1) 어느 쪽이 권위인지 미규정. 러너 문서화 의무로 넘겼으나 상호운용상 기본 우선순위 규정이 필요할 수 있음.
3. **progress-paths와 판정 자산 격리의 상호작용.** progress-paths가 격리 대상(assets) 경로를 포함하면, 검증 전 복원으로 해시가 항상 불변이 되어 stagnation 판정이 왜곡됨. → v0.4에서 §4.2.4에 "겹치지 않게 선언(SHOULD)" 지침으로 **부분 처리함**. 남은 문제: MUST 강제 또는 러너의 자동 배제(assets를 progress 해시에서 빼기)로 승격할지 v0.5 검토.
4. **Level 1.5 홀드아웃 대체의 감사 가능성.** "인간 게이트로 홀드아웃 대체"는 기술적 차단이 아니라 절차적 통제라 기계 검증 불가 — §6.1 픽스처가 Level 1.5를 어떻게 관측할지(게이트 로그 존재 확인 수준?) v0.5 스위트에서 정의 필요.
5. **잔존 정책의 부분 산출물 오염.** per-iteration 초과 "잔존" 정책에서 부분 산출물이 다음 iteration 구현 주체의 입력 상태를 오염시킬 때, 이를 로그·delta로 어떻게 드러낼지 미정.
